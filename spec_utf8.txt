1. Vue d’Ensemble du Produit
1.1 Objectif
Andar Dashboard est une plateforme web de suivi de conformité hospitalière multi-référentiel. Elle permet à un hôpital de visualiser en temps réel sa progression vers l’accréditation selon trois cadres internationaux : Accreditation Canada Diamond, JCI (8ème édition), et HAS (V2025).
Le système ingère des données via import Excel/CSV et connexion directe aux systèmes d’information hospitalier (HIS), puis calcule automatiquement les indicateurs de conformité par domaine, génère des alertes, et produit des rapports formatés pour les évaluateurs.

2. Référentiels d’Accréditation
2.1 Accreditation Canada Diamond (Qmentum Global)

6. Wireframes et Spécifications UI
6.1 Page Principale — Vue d’Ensemble
La page principale est divisée en 4 zones :
Zone 1 (haut) : 3 jauges circulaires côte à côte — score global Accreditation Canada (ex: 72%), JCI (ex: 68%), HAS (ex: 81%). Chaque jauge est colorée selon le seuil (vert/orange/rouge). Un clic ouvre la page détaillée du référentiel.
Zone 2 (milieu gauche) : Graphique en courbe — évolution du score global sur 12 mois, avec les 3 référentiels superposés. Permet de visualiser la tendance et l’impact des actions correctives.
Zone 3 (milieu droite) : Tableau des alertes actives — les 10 non-conformités les plus critiques, triées par priorité. Chaque ligne montre : domaine, standard, département, délai dépassé (en jours), responsable. Code couleur rouge/orange.
Zone 4 (bas) : Heatmap départements × domaines — une matrice montrant le score de conformité de chaque département sur chaque domaine. Permet d’identifier visuellement les zones faibles.
6.2 Page Référentiel Détaillé
Sélection du référentiel via onglets (AC | JCI | HAS). Pour chaque référentiel :
Radar chart : un axe par domaine, score 0–100%. Permet la comparaison visuelle immédiate des forces et faiblesses.
Liste déroulante par domaine : chaque domaine s’ouvre pour révéler ses standards. Chaque standard affiche : statut (conforme/partiel/non-conforme/NA), dernière évaluation, département responsable, nombre de preuves téléchargées.
Filtre par département : permet à un chef de service de voir uniquement ce qui le concerne.
Bouton « Voir correspondances » : pour chaque standard, affiche les standards équivalents dans les autres référentiels.
6.3 Page Import Données
Le flux d’import est conçu pour minimiser les erreurs :
Étape 1 : Sélection du template (dropdown avec les 8 templates définis). Le système affiche le format attendu et un lien de téléchargement du template vide.
Étape 2 : Upload du fichier (drag & drop ou sélecteur). Validation immédiate : format, colonnes manquantes, types de données.
Étape 3 : Prévisualisation des 10 premières lignes avec surlignage des erreurs en rouge. L’utilisateur peut corriger ou rejeter.
Étape 4 : Confirmation et import. Le système affiche le nombre de lignes traitées, les KPIs mis à jour, et les alertes déclenchées.
6.4 Page KPIs Temps Réel
Grille de cartes KPI, chacune affichant :
Nom du KPI et code
Valeur actuelle avec code couleur (vert/orange/rouge)
Sparkline des 6 derniers mois
Tendance (↑ ↓ →) et variation en %
Clic → modal avec historique complet, graphique détaillé, et lien vers les standards concernés

8. Sécurité et Conformité Réglementaire
8.1 Principes Non-Négociables
Aucune donnée patient nominative ne transite dans le système. Tous les identifiants sont anonymisés avant import.
Hébergement sur serveur interne CHU UM6SS ou cloud privé marocain. Aucune donnée ne quitte le territoire.
Conformité Loi 09-08 (CNDP) : déclaration préalable, consentement, droit d’accès, droit de rectification.
Chiffrement en transit (TLS 1.3) et au repos (AES-256).
Authentification JWT avec refresh tokens, expiration 24h, invalidation sur déconnexion.
Rôles et permissions : Admin, Quality, DeptHead, Viewer. Principe du moindre privilège.
Piste d’audit complète : chaque action (import, évaluation, modification) est tracée avec horodatage, utilisateur, IP.
Sauvegardes automatisées quotidiennes de la base de données.
8.2 Gestion des Appels API Externes
Si l’IA (Gemini) est utilisée pour générer des recommandations, seules des données agrégées et anonymisées sont envoyées. Les prompts ne contiennent jamais de noms de patients, de numéros de dossiers, ou de données identifiantes. À terme, l’objectif est de migrer vers un modèle local (MedGemma 27B) pour éliminer toute dépendance externe.
